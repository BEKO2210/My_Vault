  <name>Task 1: Create briefing-utils.cjs data aggregation module</name>
  <files>.agents/skills/briefing/briefing-utils.cjs</files>
  <action>
Create `.agents/skills/briefing/` directory and `briefing-utils.cjs` with the following exported functions:

1. **`gatherBriefingData(vaultRoot)`** -- Main entry point. Returns a structured object containing all data Claude needs to compose the briefing:
   - `recentChanges`: Array of notes modified in last 48 hours (default) sorted newest-first. Each entry: `{ path, name, type, folder, mtime, hoursAgo }`. Exclude templates, `.claude/`, and `06 - Atlas/` notes. Cap at 10 items, include `totalRecent` count.
   - `projects`: Active projects from `getActiveProjects(vaultRoot)` (reuse memory-utils).
   - `neglected`: Active/planned projects with mtime older than 14 days. Each: `{ path, name, status, priority, lastModified (YYYY-MM-DD), daysSinceModified }`. Sort oldest first.
   - `inboxCount`: Count of notes in `00 - Inbox/` excluding Inbox.md, Daily Notes subfolder, and templates.
   - `insights`: Top 5 insights from `parseInsights(vaultRoot)` with confidence >= 0.5.
   - `memoryContent`: Raw content of MEMORY.md (read via fs).
   - `vaultStats`: `{ totalNotes, lastScan }` from vault-index.json and scan-state.json.

2. **`getRecentChanges(vaultIndex, lookbackMs)`** -- Extracts recently modified notes from the vault index. Lookback defaults to 48 hours in ms. Returns array sorted by mtime descending.

3. **`getStaleProjects(vaultIndex, staleDays)`** -- Finds projects with status active/planned and mtime older than `staleDays` (default 14). Returns array sorted oldest-first.

**Implementation details:**
- Use `require('../scan/utils.cjs')` for `loadJson`.
- Use `require('../memory/memory-utils.cjs')` for `getActiveProjects` and `parseInsights`.
- Use Node.js built-in `fs` and `path` only.
- Follow the exact CJS pattern from existing utils: `'use strict';` at top, `module.exports = { ... }` at bottom.
- Include JSDoc comments on all exported functions matching the style in health-utils.cjs.
- The 48-hour lookback and 14-day neglect threshold should be constants at module top, not magic numbers inline.


   ```
   /briefing --days 7     # Extended lookback (7 days)
   ```

3. **What It Shows section** -- table of briefing sections:
   | Section | Source | Cap |
   |---------|--------|-----|
   | Recent Changes | vault-index.json mtimes | Top 5, "N more" summary |
   | Active Projects | memory-utils getActiveProjects | All, sorted by priority |
   | Neglected Items | Projects with stale mtime | Top 3 |
   | Inbox Status | Inbox note count | Count only |
   | Suggestions | Claude reasoning over all data | 1-2 items |

4. **Execution Flow** -- numbered steps (matching /create and /health patterns):
   1. Ensure fresh indexes: `ensureFreshIndexes('.')`
   2. Gather briefing data: `gatherBriefingData('.')`
   3. Format as calm executive briefing (Claude composes, not template)
   4. Cap output at 15-20 lines. Use "N more items" summaries for overflow.
   5. Present with clear section headers but no excessive formatting.

5. **Output Format section** -- example of what the briefing looks like:
   ```
   ## Daily Briefing -- 2026-03-07

   **Recent Changes** (last 48h)
   - Modified: [[Website Relaunch]] (project, 2h ago)
   - Created: [[Tool -- Docker]] (tool, 5h ago)
   - 3 more changes

   **Active Projects**
   - [[Website Relaunch]] -- high priority, active
   - [[API Migration]] -- medium priority, planned

   **Needs Attention**
   - [[Blog Platform]] -- active but untouched for 23 days
   - 4 notes in Inbox awaiting triage

   **Suggestions**
   - Consider running /triage to classify inbox items
   ```

7. **Utility Module** section: Reference `briefing-utils.cjs` with function signatures.

   - No active projects: Skip the projects section.
   - No neglected items: Skip the needs attention section.
